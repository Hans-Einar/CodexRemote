# Design - App Server Integration

**Status:** Draft v0.1  
**Date:** 2026-04-01

## 1) Purpose
Define how CodexRemote integrates with the local Codex surface.

## 2) Chosen Model
CodexRemote uses a bridge-side adapter abstraction.

The browser UI never talks directly to the local Codex surface.  
Instead, the bridge owns one or more adapters that normalize local integration behavior.

## 3) Adapter Contract
The bridge adapter must support these conceptual operations:

- discover sessions
- list or resolve threads for a session
- read thread history
- send a prompt
- subscribe to thread events when available

The adapter may also expose capability flags such as:

- supportsAttach
- supportsStreaming
- supportsWorkspaceHints

## 4) Mode Handling

### Mode B
Use the adapter to attach to or mirror an existing live thread when the local surface exposes a stable path for doing so.

### Mode A
Use a fallback adapter or fallback workflow to manage separate local threads while preserving the same bridge-facing API shape.

## 5) Failure Handling
If attachment or streaming is not supported:

- expose capability limits explicitly
- keep read and send flows available where possible
- degrade to polling or fallback thread management without hiding the mode change
