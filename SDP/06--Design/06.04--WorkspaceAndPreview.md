# Design - Workspace and Preview

**Status:** Draft v0.1  
**Date:** 2026-04-01

## 1) Purpose
Define the workspace browsing, preview, and editor behavior for the current implementation scope.

## 2) Chosen Model
The bridge exposes workspace metadata and raw file content.  
The browser client owns preview rendering and editor presentation.

## 3) Workspace Rules

- every requested path must resolve under the selected workspace root
- large files may be rejected or truncated behind explicit limits
- binary files are out of first scope unless later design updates add them
- supported text files may be edited and saved back through the bridge

## 4) File Payload Shape
The file-read surface should provide:

- normalized relative path
- detected content type
- text content when available
- file size
- last modified time when available

The file-write surface should provide:

- normalized relative path
- saved content length
- last modified time when available

## 5) Markdown Preview
Markdown should be rendered client-side from sanitized content.

The bridge is responsible for:

- reading the raw file safely
- reporting metadata

The UI is responsible for:

- markdown rendering
- code-block presentation
- link behavior and preview layout
- editor presentation for supported files
- explicit dirty-state and save-state handling

## 6) Current Component Direction
The current preferred UI components for this area are:

- an explorer-style tree component for workspace navigation
- Monaco-based editing for supported text files

These choices are meant to reduce custom editor and tree maintenance burden.
