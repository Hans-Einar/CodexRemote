# Design - Thread and Session Model

**Status:** Draft v0.1  
**Date:** 2026-04-01

## 1) Purpose
Define the canonical model that the rest of CodexRemote uses.

## 2) Canonical Entities

### WorkspaceRef
Represents the active project root when known.

Fields should include:

- workspaceId
- rootPath
- displayName

### SessionRef
Represents a local Codex work container.

Fields should include:

- sessionId
- workspaceId when known
- surfaceKind
- capabilities

### ThreadRef
Represents a conversation surface.

Fields should include:

- threadId
- sessionId
- mode
- attachability
- title or label

### MessageRef
Represents one ordered message in a thread.

Fields should include:

- messageId
- threadId
- role
- content
- status
- createdAt when available

## 3) Mode Semantics

- `mode = attached` means the bridge believes it is representing an existing live thread.
- `mode = mirrored` means the bridge can read or approximate an external thread but cannot guarantee direct continuation fidelity.
- `mode = fallback` means the bridge manages a separate local thread.

## 4) Event Semantics
Event payloads should use the same canonical identifiers as snapshot reads.

At minimum, the system should support:

- thread-updated
- message-added
- message-chunk
- message-completed
- connection-state-changed

## 5) Design Rule
The UI must never infer attachment fidelity from transport success alone.  
Attachment state must be explicit in the canonical model.
