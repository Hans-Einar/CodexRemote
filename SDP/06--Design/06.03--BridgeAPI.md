# Design - Bridge API

**Status:** Draft v0.1  
**Date:** 2026-04-01

## 1) Purpose
Define the stable local API that the web client uses.

## 2) API Style
Use:

- HTTP for request and response operations
- WebSocket for live events

## 3) Baseline HTTP Surface

- `GET /api/health`
- `GET /api/modes`
- `GET /api/sessions`
- `GET /api/threads?sessionId=...`
- `GET /api/threads/:threadId`
- `POST /api/threads/:threadId/messages`
- `GET /api/workspace/tree`
- `GET /api/workspace/file?path=...`

Optional later additions:

- `POST /api/fallback/threads`
- `POST /api/refresh`

## 4) WebSocket Surface
Expose a single event stream endpoint such as:

- `WS /api/events`

Event messages should carry:

- event type
- canonical identifiers
- payload body

## 5) Error Model
Errors returned to the UI should be stable and user-displayable.

The response shape should include:

- code
- message
- retriable flag
- details only when safe to expose

## 6) Security Rule
The API is local-only by default and should bind to localhost unless the user explicitly broadens exposure.
