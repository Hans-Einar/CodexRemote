# Design Analysis - Workspace and Markdown

**Status:** Draft v0.1  
**Date:** 2026-04-01

## 1) Purpose
Harden the first-scope design for workspace browsing and markdown preview.

## 2) First-Scope Constraints

- keep workspace access read-only
- scope filesystem access to the active project root
- support text and markdown before broader file-type handling
- avoid embedding renderer-specific assumptions in the bridge where possible

## 3) Direction
The bridge should expose raw file content and normalized metadata, while the browser client owns markdown rendering.

Benefits:

- cleaner separation between file access and presentation
- easier UI iteration
- reduced coupling to any one markdown renderer

## 4) Preferred Decision
Start with:

- directory listing
- text-file read
- markdown preview
- path normalization and project-boundary enforcement

Later enhancements may include:

- image preview
- syntax highlighting
- search

## 5) Open Risks

- workspace-root discovery may be indirect or user-assisted at first
- large-file handling needs explicit limits
- markdown rendering must avoid unsafe HTML execution by default
