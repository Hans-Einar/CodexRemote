# Design Analysis - Web UI

**Status:** Draft v0.1  
**Date:** 2026-04-01

## 1) Purpose
Harden the baseline UI structure for chat, workspace browsing, and mode visibility.

## 2) UI Problems To Solve

- make Mode B versus Mode A status obvious
- make connection failures understandable
- support both desktop and mobile usage
- let the user switch quickly between chat context and file context

## 3) Direction
The initial UI should be simple and structural rather than visually ambitious.

Recommended first layout:

- thread or session list
- conversation view
- workspace and preview panel

On narrow screens, these areas should become switchable panels rather than fixed columns.

## 4) Preferred Decision
The UI should surface:

- connection state
- current mode
- selected workspace
- selected thread
- recent error or degraded-state messaging

## 5) Open Risks

- narrow-screen layouts may need different navigation patterns than desktop
- overly optimistic mode labels could mislead the user about actual attachment fidelity
- live-update latency needs visible feedback to preserve trust
