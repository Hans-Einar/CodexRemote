# Design Analysis - Session and Thread Sync

**Status:** Draft v0.1  
**Date:** 2026-04-01

## 1) Purpose
Harden the model for sessions, threads, workspace identity, and Mode B attachment behavior.

## 2) Main Problem
The user experience depends on attaching to the same work context already active in Codex Desktop or VS Code, but the exact relationship between workspace, session, and thread is not yet proven.

## 3) Direction
CodexRemote should maintain a canonical model with explicit attachability state.

At minimum, the model should represent:

- workspace reference
- session reference
- thread reference
- operating mode
- attachability or confidence

## 4) Preferred Decision
Model attachment as best-effort and explicit.

That means:

- do not silently treat mirrored or fallback threads as the same thing
- expose whether a thread is directly attached, mirrored, or managed fallback
- keep workspace association optional when the local surface does not expose it reliably

## 5) Open Risks

- the local surface may expose conversation history without exposing a stable attach point
- one workspace may map to multiple sessions or no session at all
- cross-surface sync timing may be eventual rather than immediate
