# Design Analysis - Bridge Runtime

**Status:** Draft v0.1  
**Date:** 2026-04-01

## 1) Purpose
Analyze the role of the Node bridge as the boundary between the local Codex surface, the filesystem, and the browser UI.

## 2) Runtime Responsibilities
The bridge should own:

- protocol adapter loading
- canonical session and thread orchestration
- workspace path resolution
- HTTP and WebSocket serving
- health and error reporting

## 3) Direction
The bridge should be the authoritative local integration layer, but not the source of truth for conversation state.

This implies:

- the adapter remains the source for thread and message data
- the bridge may cache transient state for responsiveness
- cached data must remain disposable and reconstructable

## 4) Preferred Decision
Use HTTP for query-style operations and WebSocket for live events where helpful.

This keeps:

- browser integration straightforward
- non-streaming flows simple
- streaming or polling upgrades possible without changing the UI contract shape too aggressively

## 5) Open Risks

- long-lived subscriptions may be fragile if the local Codex surface is short-lived
- workspace and conversation events may have different latency characteristics
- error translation may hide useful protocol detail unless logging is designed carefully
