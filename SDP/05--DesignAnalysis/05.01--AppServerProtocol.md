# Design Analysis - App Server Protocol

**Status:** Draft v0.1  
**Date:** 2026-04-01

## 1) Purpose
Analyze how CodexRemote should integrate with the local Codex app-server or equivalent local surface.

## 2) Key Questions

- Which transport is actually available: stdio, HTTP, WebSocket, or another local IPC path?
- Which identifiers are stable enough for session and thread attachment?
- What message-send and message-read operations exist?
- Is streaming a first-class protocol feature or only an implementation detail?

## 3) Direction
The browser client should never depend directly on the protocol shape.

CodexRemote should introduce a bridge-side adapter contract that can normalize:

- session discovery
- thread discovery
- message history read
- message send
- event subscribe

## 4) Preferred Decision
Treat the protocol as replaceable and partially undocumented.

This implies:

- thin protocol adapters
- a canonical session and thread model above the adapter
- a polling fallback even if streaming is later supported

## 5) Open Risks

- auth may be tied to runtime state that the bridge can only reuse indirectly
- thread attach may require identifiers not exposed through a stable API
- protocol behavior may differ between desktop and VS Code surfaces
